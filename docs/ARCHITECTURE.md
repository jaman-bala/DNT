# Архитектура проекта DNT

В основу проекта положена Enterprise-архитектура, ориентированная на масштабируемость, легкое тестирование и асинхронную производительность.

## 🏗 Основные слои

Проект разделен на три основных логических слоя:

1.  **Controller (API Layer)**: `apps/*/controllers/`
    *   Отвечает за прием HTTP-запросов и валидацию входных данных через Pydantic (DTO).
    *   Не содержит бизнес-логики.
    *   Взаимодействует с сервисами через DI контейнер.
2.  **Service (Business Logic Layer)**: `apps/*/services/`
    *   Содержит основную логику приложения.
    *   Все методы асинхронные (`async def`).
    *   Не зависит от HTTP-протокола.
3.  **Repository/Model (Data Layer)**: `apps/*/models/` и `apps/*/repository/`
    *   Отвечает за структуру данных и прямое взаимодействие с базой данных.
    *   Использует `BaseModel` для обеспечения единого стандарта полей (UUID, timestamps, soft-delete).

## 💉 Внедрение зависимостей (Dependency Injection)

Для управления зависимостями используется простой и эффективный DI контейнер в `src/config/container.py`.

### Зачем это нужно?
*   **Тестируемость**: Легко подменять реальные сервисы на Mock-объекты в тестах.
*   **Связность**: Сервисы не создают друг друга напрямую, что уменьшает жесткую связь между модулями.
*   **Singleton по умолчанию**: Контейнер гарантирует наличие только одного инстанса каждого сервиса.

### Использование:
```python
from config.container import container

@router.post("/register")
async def register(request, data: UserRequestDTO):
    user = await container.user_service.create_user(data)
    return user
```

## 🛠 Базовые классы (Base Classes)

Для минимизации дублирования кода используются базовые абстракции в `src/config/base/`:

### `BaseModel`
Все модели проекта наследуются от `BaseModel`, что автоматически добавляет:
*   `id`: UUID v7 (сортируемый UUID для лучшей производительности индексов).
*   `created_at` / `updated_at`: Автоматические метки времени.
*   `is_active`: Флаг активности.
*   `is_deleted`: Поддержка Soft Delete (мягкого удаления).
*   `BaseManager`: Автоматическая фильтрация удаленных записей (`is_deleted=False`).

### `BaseService`
Базовый класс для всех сервисов, обеспечивающий единую структуру инициализации.

## ⚡ Асинхронность (Async)

Проект полностью асинхронный. Основные правила:
*   Все эндпоинты в контроллерах помечены как `async def`.
*   Все методы в сервисах `async def`.
*   Использование асинхронных ORM методов Django (например, `ainit`, `asave`, `afirst`, `acreate`).
*   Работа с внешними I/O (S3, Redis, External APIs) только через асинхронные драйверы.

## 🔒 Безопасность

*   **JWT с Cookies**: Access-токен передается в заголовке, а Refresh-токен хранится в HTTP-only куке для защиты от XSS атак.
*   **Pydantic V2**: Строгая типизация и валидация всех входящих и исходящих данных.
*   **Custom Types**: Использование специальных типов в `config/types.py` для валидации телефонов, паролей и имен.
